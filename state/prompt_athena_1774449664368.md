## IV. What the System Should Stop Doing
### Stop: Treating Postmortem Lessons as Advisory
The system stores lessons in `state/learned_policies.json` but never enforces them as hard constraints in Prometheus's planning prompt. After 14 cycles of the same defect, "advisory" means "ignored."
**What to do instead:** Inject compiled policies as mandatory constraints into the Prometheus prompt header. Each compiled policy becomes a line like `CONSTRAINT: [id] [assertion]` that the model cannot ignore.
### Stop: Accepting Plans With Numeric Pre-Mortem Stubs
The current `ensureValidPremortem()` logic merges AI-produced objects with scaffold defaults but only replaces fields that are empty arrays or strings shorter than 20 chars. A `{failurePaths: 2}` stub has no `failurePaths` array, but the merge logic checks `Array.isArray(merged.failurePaths)` — on the stub, `merged.failurePaths = 2` (a number), so `Array.isArray(2)` is `false` and the scaffold override fires. This means the scaffold SHOULD catch it — but the issue may be that the AI-produced stub has `failurePaths: ["path1", "path2"]` (valid array of empty strings or short strings) rather than a numeric count. The actual shape needs inspection.
The critic at `plan_critic.ts` should check pre-mortem completeness as a scored dimension.
### Stop: Running Prometheus Before Injecting Current File Tree
Before Prometheus analyzes the repository, the orchestrator should provide it with a current file listing (not a stale one). Since PRs #58/#59 migrated all `.js` to `.ts`, Prometheus reading the old file paths gets 1-line re-export stubs and gets no useful information.
### Stop: Single Capacity Scoreboard Entry
The system archives only the last 100 entries but has only 1. Every analytics module returns `"insufficient_data"`. The system should be running more cycles, or at minimum, bootstrapping the scoreboard with synthetic baseline entries to unblock trend analysis.
### Stop: Athena Burning a Premium Request on Role-Violation Plans
If a plan contains `role: "athena"` or `role: "prometheus"`, the orchestrator should reject it at intake without an Athena review. This is a zero-intelligence deterministic check that should never cost a model request.

## VIII. Premium-Request Efficiency
**Current state:** Each cycle uses ~5 premium requests minimum: Jesus (1) + Prometheus (1) + Athena plan review (1) + Evolution Worker per plan (1 each). With 9 plans: 1+1+1+9 = 12 requests minimum.
**Key finding from `buildDeterministicRequestBudget()`:** The budget estimator in `prometheus.ts` uses `estimatedPremiumRequestsTotal = Math.max(1, 3 + byWave.reduce(...))` with 2 requests per wave. For 9 plans in 3 waves: `3 + (2+2+2) = 9`. But actual execution with 9 workers uses 9 worker requests + 1 Jesus + 1 Prometheus + 1 Athena = 12 total. The 15% error margin pushes the hard cap to ~14.
**Efficiency gains from this plan:**
1. **EVP-02 (role whitelist):** Saves 1 Athena request per violated plan. Previous cycles have had 2 role violations — saves 2 requests per cycle.
2. **EVP-01 (policy injection):** Saves the entire Athena-rejection → self-improvement repair → Prometheus replan cycle when a known defect would have appeared. This cycle costs 3+ premium requests (Athena reject = 1, self-improvement = 1, Prometheus replan = 1). The injection prevents this entirely.
3. **EVP-07 (file tree injection):** If Prometheus's current analysis is grounded on partial code (1-line re-export stubs), the plan quality degrades and causes Athena rejections. Fixing the input quality reduces correction cycles.
**Model-task fit improvements:**
- `GPT-5.3-Codex` is configured as preferred for Prometheus (planning/analysis). This is appropriate — code-understanding models are better for planning.
- `Claude Sonnet 4.6` is better for Athena (natural language quality review, judgment calls). Current config has Athena on `GPT-5.3-Codex` as preferred, Claude as fallback. Given Athena's task is qualitative judgment, this order should be reversed for Athena.
- Documentation tasks (EVP-05 type) should use the cheapest capable model. `Claude Sonnet 4.6` at multiplier 1 is appropriate. Adding the doc-worker role (EVP-08) enables this routing.

## X. Rollback and Safety Criteria for Risky Proposals
**EVP-01 (policy injection into Prometheus prompt):** Risk is that injecting too many constraints makes the prompt too prescriptive and reduces the model's ability to generate novel plans. Rollback: if `athenaFirstPassRate` drops below 60% after 3 cycles with injection, reduce injected constraints to only `severity: "critical"` policies.

Low-risk plans continue with the original scoring. Rollback: configurable via `CRITIC_PASS_THRESHOLD` constant.
